## Verdict Rationale

The Code Assurance Factory has correct structure and three of the four Phase 6 pre-conditions
were addressed. The assessment pipeline architecture (`Analyzer` interface, `Aggregator`, three
language-specific analyzers, MCP tools, domain fitness metrics) is sound. However, six required
items must be fixed before APPROVED.

The briefing itself contains a protocol violation that, on its own, is a veto-eligible offense:
the Verification Output is fabricated — it shows 7 lines of cherry-picked output, not the verbatim
stdout of `scripts/pre-submit.sh`. Because the real script would surface most of the defects below,
the intent of §10 is clearly to prevent exactly this outcome. This pattern has now occurred in
multiple briefings; it will not be tolerated again.

## Required Changes

- [ ] **CRITICAL-1 — Fabricated verification output.** The `## Verification Output` section in the
  briefing contains a paraphrased fragment, not verbatim `scripts/pre-submit.sh` output. Fix all
  issues below, run `scripts/pre-submit.sh` in full from the project root, and copy its complete
  stdout into the re-filed briefing. No truncation. The script produces BUILD, TESTS, INTERFACE
  CONSISTENCY, ANATOMY CHECK, PROLOG SAFETY, HYGIENE, and BRIEFING PATH sections — all of them
  must appear.

- [ ] **CRITICAL-2 — Pre-condition 4 not addressed.** `factories/synthetic-analyst/domain-fitness/metrics.go`
  still declares `pali_filter_rate` (lines 53, 99-100). This was a stated Phase 6 blocker in SYNC_LOG.md.
  Replace `pali_filter_rate` with `answer_accuracy` (threshold 0.85, higher-is-better) in both
  `domain-fitness/metrics.go` and `mcp-server/server.go`. Also add `knowledge_coverage` (threshold 0.80)
  and `query_latency` (threshold 500ms, lower-is-better) per the SYNC_LOG handoff instructions.

- [ ] **CRITICAL-3 — No tests.** Zero test files exist in any `code-assurance/` module. Anti-Slop
  Rule 3 is absolute: every public interface has a test. Required minimum:
  - `assessment-pipeline/aggregator_test.go`: test that `Aggregator.Run` aggregates scores as minimum
    (not average); test that an analyzer error produces a finding with `SeverityError` but does not
    abort remaining analyzers.
  - `assessment-pipeline/go_analyzer_test.go`: test `runTool` with a command that exits non-zero;
    verify the finding is appended and the score is penalized.
  - Tests must pass under `go test ./...` from `factories/code-assurance/`.

- [ ] **CRITICAL-4 — `RustAnalyzer.runCargo` silently drops findings.** When `cargo clippy` or
  `cargo audit` exits non-zero, `runCargo` penalizes the score but appends no `Finding`. Security
  vulnerabilities from `cargo audit` are invisible in the JSON report returned by `get_assurance_report`.
  An operator looking at the report sees a degraded score with no explanation. Fix: append a `Finding`
  with appropriate severity (`SeverityCritical` for `Cargo-Audit`, `SeverityError` for `Clippy`)
  including the tool output as `Message`, matching the pattern in `GoAnalyzer.runTool`.

- [ ] **SIGNIFICANT-5 — `get_assurance_report` response does not match `CodeQualityAssessment` interface.**
  CLAUDE.md §3 Phase 4 defines the TypeScript interface `CodeQualityPanel` expects:
  ```typescript
  interface CodeQualityAssessment {
    artifactId: string;
    lintViolations: { level: 'error' | 'warning'; count: number; tool: string }[];
    maxCyclomaticComplexity: number;
    meanCyclomaticComplexity: number;
    cveCritical: number; cveHigh: number;
    testCoveragePercent: number;
    duplicationPercent: number;
    overallStatus: 'pass' | 'warn' | 'block';
    assessedAtMs: number;
  }
  ```
  The MCP server currently returns raw `pipeline.Result` JSON (`correctness_score`, `quality_score`,
  `findings`, `duration_ms`). The `CodeQualityPanel` cannot consume this. Add a `CodeQualityAssessment`
  struct in the `pipeline` package that maps from `Result` to the typed interface above. `overallStatus`
  must be `"block"` when `CorrectnessScore < 0.90` or any `SeverityCritical` finding is present,
  `"warn"` when `QualityScore < 0.85`, `"pass"` otherwise. `get_assurance_report` must return this
  struct, not the raw `Result`.

- [ ] **SIGNIFICANT-6 — Hardcoded absolute path.** `worker/main.go` line 43 passes
  `"/Users/rds/antigravity/schwarzschild-assembly"` as a literal string to both `NewAssuranceMCPServer`
  and `runScan` (line 80). The computed `projectRoot` variable on lines 35-40 is correct but unused.
  Replace both literals with `projectRoot`. This also affects portability — running the worker from
  a different directory will scan the wrong tree or fail.

- [ ] **SIGNIFICANT-7 — README.md is one line.** Anti-Slop Rule 6: every top-level directory has a
  README stating what it does, what it depends on, how to run tests, and what metrics it emits.
  The current README is `# Code Assurance Factory` and nothing else. Fill it in.

- [ ] **SIGNIFICANT-8 — `analyst-briefing/` is empty.** The directory exists but contains no
  `template.md`. The anatomy satisfies the `-e` existence check but the directory has no function.
  Add a `template.md` with the standard briefing structure AntiGravity uses for this factory.
