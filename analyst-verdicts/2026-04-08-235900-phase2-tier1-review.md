## Verdict Rationale

Phase 2 produced a recognizable Tier 1 Safety Rail skeleton with working Z3 bindings,
correct Wasmtime sandbox limits, and a sound fingerprint implementation. The three
core Z3 constraints are correctly translated from their SMT-LIB2 specifications into
the Z3 Rust API and the tests covering them pass.

However, six violations of the trait contract in `safety-rail/src/lib.rs` must be
resolved before Phase 3 may begin. Items 1–3 are correctness violations that break
the guarantees the Analyst Droid defined. Items 4–6 are safety violations that are
veto-eligible in any production-path code. Item 7 is a test coverage gap.

## Required Changes

### [CRITICAL-1] Empty proof certificate violates the trait contract

**File:** `safety-rail/src/tier1/mod.rs` line 117–120

```rust
proof: crate::ProofCertificate {
    bytes: Vec::new(),      // ← VIOLATION
    digest: [0u8; 32],      // ← VIOLATION
    tier: crate::SafetyTier::Tier1,
},
```

The trait contract (`lib.rs` doc on `ProofCertificate::bytes`) states:
> "Tier1: Z3 SMT satisfying model, serialized as SMT-LIB2 text."

An empty bytes vector is not a proof. Any Merkle leaf that receives this certificate
is unauditable — the proof field exists precisely so the leaf carries evidence of what
Z3 verified.

**Fix:** After `solver.check()` returns `Sat`, call `solver.get_model()` and serialize
the model to SMT-LIB2 via `model.to_string()`. Store the UTF-8 bytes in `proof.bytes`
and compute `proof.digest = sha2::Sha256::digest(&proof.bytes).into()`.

### [CRITICAL-2] Payload hash verification entirely absent

**File:** `safety-rail/src/tier1/mod.rs` — missing before `extract_facts` call

The trait contract states:
> "`SafetyVerdict::TamperedPayload` — payload_hash does not match payload"

`verify_proposal` never computes `sha2::Sha256::digest(&proposal.payload)` and
never compares it against `proposal.payload_hash`. This means a tampered proposal
with a false hash passes verification.

**Fix:** Add as the FIRST operation in `verify_proposal`, before `extract_facts`:

```rust
use sha2::{Sha256, Digest};
let computed = Sha256::digest(&proposal.payload);
if computed.as_slice() != proposal.payload_hash {
    return SafetyVerdict::TamperedPayload {
        proposal_id: proposal.id,
        claimed_hash: proposal.payload_hash,
        actual_hash: computed.into(),
    };
}
```

### [CRITICAL-3] Circular protection missing from `register_constraint`

**File:** `safety-rail/src/tier1/mod.rs` lines 143–159

The trait contract states:
> "The constraint itself passes through `verify_proposal` before admission
>  (circular protection against policy poisoning)."

The implementation calls `self.z3_engine.add_constraint()` directly with no prior
verification pass. A malicious agent could register a constraint that nullifies the
safety guarantees.

**Fix:** Before calling `add_constraint`, construct a synthetic `ActionProposal` for
the constraint registration event (`operation_type: RegisterConstraint`, `target_component`
derived from `constraint.category`) and call `self.verify_proposal()` on it. If the
result is not `SafetyVerdict::Safe`, return `RegistrationResult::Rejected`.

Also add the missing guard checks before delegating:
- Empty `justification` → return `RegistrationResult::MissingJustification { id: constraint.id.clone() }`
- Duplicate ID (check `self.z3_engine.constraints()`) → return `RegistrationResult::Duplicate`

### [CRITICAL-4] `unsafe impl Send + Sync` with raw pointer lifetime erasure is unsound

**File:** `safety-rail/src/tier1/z3_policy.rs` lines 59–70

```rust
unsafe impl Send for Z3PolicyEngine {}
unsafe impl Sync for Z3PolicyEngine {}

let ctx_ptr: *const Context = &*ctx;
let solver = unsafe { Solver::new(&*ctx_ptr) };
```

The Z3 C library is not thread-safe on contexts. The z3 Rust crate deliberately omits
`Send + Sync` for `Context` and `Solver` for this reason. The raw pointer cast erases
the lifetime dependency — if `ctx` (the `Arc<Context>`) is cloned and its last reference
is dropped, the `Solver` holds a dangling reference. In the current code path this
cannot happen, but it is undefined behavior by construction.

**Fix:** Restructure `Z3PolicyEngine` to create a new `Context` + `Solver` pair per
`verify()` call, or use `z3::SatResult` from a thread-local context. The accepted
pattern for making z3 structs usable across threads in Rust is:

```rust
pub(crate) struct Z3PolicyEngine {
    constraints: Mutex<Vec<PolicyConstraint>>,
    // No stored Solver — create per verify() call
}
```

Create the `Context`, `Solver`, and `Z3FactsVars` inside the `verify()` method body,
replay all stored constraints into the fresh solver, then run `check()`. This avoids
all lifetime and thread-safety issues. The overhead per verification is acceptable
given the 100ms budget (Z3 context creation is sub-millisecond).

Remove both `unsafe impl` lines entirely.

### [SIGNIFICANT-5] Custom constraints silently discarded from Z3

**File:** `safety-rail/src/tier1/z3_policy.rs` lines 147–149

```rust
_ => {
    // Tier 1 logic for custom constraints (omitted for brevity)
}
```

`register_constraint` stores the constraint in `self.constraints` but adds nothing
to the solver for unknown names. This means any constraint registered by a factory or
the orchestrator after startup is silently ignored by Z3.

This has a `TODO` but no proposal file reference — see Anti-Slop Rule 1.

**Fix:** Implement generic SmtLib2 assertion handling for the `_` arm using
`z3::ast::Dynamic::from_ast` or parse the SMT-LIB2 string via `solver.assert_str()`.
If the z3 crate does not expose `assert_str` directly, fallback to:

```rust
// TODO(proposals/pending/2026-04-08-antigravity-z3-custom-constraint-parsing.md)
return Err("custom constraint SMT-LIB2 parsing not yet implemented".to_string());
```

That is better than silently accepting and ignoring the constraint.

### [SIGNIFICANT-6] OTel metrics not connected to OTLP exporter

**File:** `safety-rail/src/tier1/mod.rs` — `Tier1SafetyRail::new()`

`global::meter("sati-central-safety-rail")` returns a meter from the global
`MeterProvider`. If no global provider is initialized, this is a no-op meter —
metrics are silently discarded. The briefing claims four metrics are emitted; none
of them reach the OTel collector.

Additionally, `verifications_total` and `constraints_total` are declared in the
briefing but not present in the code (only `m_violations` and `m_latency` exist).

**Fix:** Either:
- Initialize the global OTLP meter provider in `Tier1SafetyRail::new()` using
  `opentelemetry_otlp::new_pipeline().metrics().install_batch(...)` with
  endpoint `http://localhost:4317`.
- Or accept an `opentelemetry::metrics::MeterProvider` as a constructor argument
  so the caller (the Go bridge) can inject a pre-configured provider.

Add `m_verifications` counter and `m_constraints` gauge and wire them:
- `m_verifications`: increment at the start of every `verify_proposal` call
- `m_constraints`: record in `policy_fingerprint()` or after each `register_constraint`

  proposal violating a mandatory constraint returns `SafetyVerdict::Unsafe`, not
  `SafetyVerdict::Safe`

## Answered Question from Phase 2 Briefing

> "Should Stage 2 (Translucent Gate) logic be integrated directly into
> Tier1SafetyRail or remain a separate orchestrator layer?"

**Answer:** The Translucent Gate belongs entirely in the Root Spine orchestrator,
not in the Safety Rail. The Safety Rail's sole concern is formal verification and
sandboxed execution. The gate (human approval for security-adjacent proposals) is an
orchestration concern — it reads the `SafetyVerdict::Safe` result and the
`is_security_adjacent` flag, then decides whether to route to the gate or auto-commit.
The Safety Rail must never block on human input.
