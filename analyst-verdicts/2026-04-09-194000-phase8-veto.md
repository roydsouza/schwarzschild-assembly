## Required Changes

- [ ] **REGRESSION-1 — Root-spine test regression. Must fix before filing.**
  AntiGravity's changes to `root-spine/internal/grpc/server.go` broke two lifecycle tests that
  were passing at Phase 7 APPROVED:

  **`Fail_INTAKE_to_DESIGN_(not_finalized)` → got OK, expected FailedPrecondition:**
  The INTAKE→DESIGN gate (`is_finalized` check) has been removed from `AdvanceLifecycle`.
  The switch statement now starts at `case LIFECYCLE_DESIGN` — the INTAKE case is gone.
  Restore it: when `currentState == LIFECYCLE_INTAKE`, fetch the spec and check `IsFinalized`.
  If false, return `codes.FailedPrecondition("spec must be finalized via finalize_spec before
  entering DESIGN")`.

  **`Fail_Unimplemented_Phase_8_Gate` → got FailedPrecondition, expected Unimplemented:**
  The DESIGN case now scans the filesystem for verdict files and returns `FailedPrecondition`
  when none match. The Phase 7 APPROVED code returned `codes.Unimplemented` for all states
  beyond INTAKE. The filesystem scan is fragile (searches for "phase7-approved" as a fallback)
  and not the right approach for Phase 8. Revert the DESIGN→SCAFFOLD gate to
  `codes.Unimplemented` with a `[PHASE-8]` comment — or implement it properly by querying the
  verdict store, not the filesystem. Do not leave a half-implemented gate that breaks the
  existing test contract.

- [ ] **CRITICAL-2 — Prolog plunit tests require live MCP host.**
  `test_safe_assert.pl` calls `safe_assert(test_skill(42))`, which calls
  `merkle_bridge:merkle_commit`, which calls the MCP tool `submit_skill_proposal` on `:8082`.
  When `:8082` is not running, all three tests fail with a connection error. Tests must pass
  in an offline environment without external services.

  Fix: add a test-mode flag to `safety_bridge.pl`. When running under plunit (detectable via
  `current_prolog_flag(test_mode, true)` or a module-level override predicate), bypass the
  MCP call and perform only the CHR constraint check locally. This validates the policy layer
  (which is what the tests are actually testing) without requiring a live server.

  Alternatively, define a mock `merkle_commit/3` in the test file that succeeds for
  safe clauses and fails for unsafe ones, loaded before `safety_bridge`:
  ```prolog
  :- module(merkle_bridge, [merkle_commit/3]).
  merkle_commit(_, _, mock_proof).   % always succeeds for test harness
  ```
  The CHR constraints in `constraints.chr` handle the unsafe-clause rejection — if they work
  correctly, tests 2 and 3 (`unsafe_assertion_banned_predicate`, `unsafe_assertion_shell`)
  will throw `safety_violation` before the MCP call is ever made.
