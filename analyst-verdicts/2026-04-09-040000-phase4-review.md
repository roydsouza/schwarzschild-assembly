## Required Changes

- [ ] **CRITICAL-1 — Wrong request type in `denyProposal`**

  `useOrchestrator.ts` lines 159–162:
  ```typescript
  const req = new ApprovalRequest();
  req.setProposalId(id);
  // ...
  await grpcRef.current.vetoAction(req, {});
  ```
  The generated client signature is `vetoAction(request: VetoRequest, ...)`. `ApprovalRequest` has a different protobuf field layout (`proposal_id` at field 1, `operator_signature` at field 2). `VetoRequest` has `proposal_id` at field 1 but also `vetoed_by`, `rationale`, `required_changes`, `veto_timestamp_ms` at fields 2–5. Passing `ApprovalRequest` encodes the wrong binary payload. At best the server deserializes a `VetoRequest` with an empty `vetoed_by`; at worst the binary frame is rejected. Either way, the veto path is corrupted.

  Fix: Replace `new ApprovalRequest()` with `new VetoRequest()` and import `VetoRequest` from `@/types/orchestrator_pb`. Set `req.setProposalId(id)` — other VetoRequest fields may be empty for now. Update the `denyProposal` type annotation accordingly.

- [ ] **CRITICAL-2 — gRPC-Web proxy absent**

  `GRPC_URL = 'http://localhost:50051'`. The Go server on `:50051` is a raw gRPC server (`grpc.NewServer()`), not a gRPC-Web server. Browsers cannot speak raw gRPC: it requires HTTP/2 with binary framing, no CORS support, and the `Content-Type: application/grpc` media type that browsers reject. The `grpc-web` protocol uses `application/grpc-web+proto` with a base64 envelope and requires the server to strip the envelope before forwarding to the gRPC handler.

  The fix is one of:
  - **Option A (preferred, no Envoy dependency):** Wrap the existing gRPC server with `github.com/improbable-eng/grpc-web/go/grpcweb`. Expose a second HTTP listener (e.g., `:8081`) that serves the wrapped handler. Update `GRPC_URL` in the hook to `http://localhost:8081`.
  - **Option B:** Add an Envoy sidecar config in `root-spine/deployments/envoy.yaml` that proxies `:8081` → `:50051` with gRPC-Web transcoding. Document in root-spine README.

  Without this fix, every `approveAction` and `vetoAction` call fails with a network error in the browser. The Translucent Gate cannot approve or deny anything.

## Fitness Vector Impact Assessment

- **Safety compliance:** DEGRADED. `denyProposal` sends wrong binary payload to `vetoAction`. Operator cannot reliably veto a flagged proposal via the UI. This violates the gate's primary safety function. Weight 0.30 — this alone prevents auto-approval under any circumstance.
- **Audit integrity:** DEGRADED (conditionally). If `vetoAction` silently accepts malformed requests, veto events may be written to the Merkle log with empty `vetoed_by` fields, corrupting the audit trail for those decisions.
- **Dhamma alignment:** Unaffected.
- **System performance:** Unaffected.
- **Operational cost:** Unaffected.

## Safety Rail Implications

No changes to the Safety Rail trait contract. The gRPC-Web proxy issue does not affect the safety verification pipeline (proposals are submitted via the Root Spine, not directly from the browser). However, the inability to approve or deny from the UI means security-adjacent proposals stall indefinitely at `PENDING_APPROVAL` — the gate is effectively frozen.

The `"SIGNED_BY_OPERATOR"` hardcoded signature is acknowledged as a placeholder and is acceptable scope for Phase 4. A Phase 5 proposal should define the cryptographic signature scheme (Ed25519 over the proposal hash with an operator key stored in macOS Keychain). This is not blocking Phase 4 APPROVED once the two above items are resolved.
