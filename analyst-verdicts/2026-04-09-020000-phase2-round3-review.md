### Required Changes — Must Fix Before Phase 3

**REGRESSION-5 — `verify()` unknown constraint arm has been silently dropped again.**

In round 1, this was the SIGNIFICANT-5 fix: the `_` arm returned `Err(...)`. That explicit error was present in the round 2 submission. In this round 3 submission it has been reverted to a silent skip:

```rust
// Current (round 3) — WRONG:
_ => {
    // Skip constraints we don't know how to verify yet.
}

// Round 2 — CORRECT:
_ => {
    return Err(format!("Unknown constraint type not yet implemented: {}", constraint.name));
}
```

The comment's reasoning is incorrect: *"We do NOT return an error here because that would block admission of any custom constraint."* It does not block admission — `add_constraint` runs in `register_constraint` *before* `verify()` is called per proposal. The silent-skip defect is: a custom constraint that passes `add_constraint`'s justification/duplicate guards gets stored in the constraint list, changes the `policy_fingerprint()`, but is *never evaluated by Z3*. This creates a permanent divergence between what the fingerprint attests and what Z3 actually enforces. That is a security defect in the audit trail.

The correct behavior is the round 2 behavior: return `Err` for unknown constraints during `verify()`. This means proposals that would be evaluated under an unimplemented constraint fail closed rather than pass silently. When custom constraint evaluation is implemented, the `Err` arm is replaced with real logic. Until then, unknown constraint names are an operational error that must surface.

Restore the `Err` return.

**NEW-1 — `runtime::Tokio` in a synchronous constructor will panic from CGO.**

`mod.rs` line 40:
```rust
let provider = opentelemetry_otlp::new_pipeline()
    .metrics(runtime::Tokio)  // ← requires active Tokio runtime
    .with_exporter(exporter)
    .build()
    .map_err(|e| ...)?;
```

`Tier1SafetyRail::new()` is a synchronous `fn`. The contract_compliance tests use `#[tokio::test]` which provides a runtime, so the tests pass. But in production, `safety_rail_new()` is called from Go/CGO — a C thread with no Tokio runtime. `runtime::Tokio` calls `Handle::current()` internally, which panics with `"no currently running executor"`. A panic across an `extern "C"` boundary is undefined behavior.

**Required fix:** Accept the `SdkMeterProvider` as an optional constructor parameter instead of building it internally:

```rust
pub fn new(meter_provider: Option<opentelemetry_sdk::metrics::SdkMeterProvider>) -> Result<Self, String> {
    if let Some(provider) = meter_provider {
        opentelemetry::global::set_meter_provider(provider);
    }
    // rest of new() unchanged — global::meter() works if provider is set, 
    // falls back to no-op if not. No panic.
    ...
}
```

The CGO bridge (`safety_rail_new()` in c_api.rs) calls `Tier1SafetyRail::new(None)` — no-op metrics in the bridge, which is acceptable since OTel in a CGO context requires a different initialization path anyway (the Go process owns the OTel SDK).

The contract_compliance tests construct a provider and pass it in, or pass `None` for tests that don't need metrics.

Update `c_api.rs` `safety_rail_new()` to call `Tier1SafetyRail::new(None)`.
