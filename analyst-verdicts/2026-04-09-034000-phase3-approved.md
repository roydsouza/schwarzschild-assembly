### Notes for Phase 4 — Not Blocking This Approval

**`SaveMerkleLeaf` and `UpdateProposalVerdict` errors are silently discarded.**
In `SubmitProposal`, `ApproveAction`, and `VetoAction`, the error returns from these DB calls are discarded. If a write fails, the in-memory Merkle tree and the database diverge — the next restart will replay fewer leaves and produce a different root. This must be fixed in Phase 4: either propagate the error (return `codes.Internal`) or at minimum log it as a structured OTel event so the divergence is observable.

**`TestVerifyProposal_Safe` uses an invalid payload schema.**
The payload `{"operation_type": "read", "resource": "..."}` does not parse as a `ProposalPayload` — `"read"` is not a valid `OperationType` and `resource` is not a known field. The safety rail returns `Unsafe` (facts extraction failure), so `result.IsSafe = false`. The test only asserts the absence of a tampering error, not that the proposal was safe. Rename it to `TestVerifyProposal_FactsExtractionFailure` or fix the payload to use a valid schema with a correct operation type.

**MCP host is still an empty directory.**
`internal/mcp/` exists but contains no files. The original Phase 3 spec required an MCP host with stdio and Streamable HTTP transports. This was not included in the CONDITIONAL required items and is not blocking Phase 3 APPROVED, but it must be implemented before Phase 5 (Synthetic Analyst Factory), which depends on the MCP host to expose the orchestrator to factory agents.

**UNSAFE verdicts do not produce Merkle leaves.**
Only `SAFE` verdicts (and gate decisions) are committed to the Merkle log. Per CLAUDE.md Section 6, every significant system event is a leaf. A safety violation is a significant event — it should produce a `SafetyViolation` leaf. This ensures the audit log captures both approvals and rejections, making it tamper-evident in both directions.
