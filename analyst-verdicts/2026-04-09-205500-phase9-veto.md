### Ground 2 — Four Substantive Code Failures

**CRITICAL-1: No Pengines sandbox evaluation in `improve_if_slow`.**

CLAUDE.md §3 Phase 8 specifies:
```prolog
evaluate_candidate(OldClause, NewClause, Verdict) :-
    pengine_create([src_list([NewClause]), sandbox(true), time_limit(5)], PengineId),
    run_test_set(PengineId, TestSet, NewResults),
    run_test_set_local(OldClause, TestSet, OldResults),
    compare_results(OldResults, NewResults, Verdict).
```
"A candidate clause only proceeds to `safe_assert/1` if `Verdict = improved` or
`Verdict = equivalent`. Regressions are discarded."

The current `improve_if_slow/2` skips this entirely and calls `safe_assert(NewClause)`
directly after `check_invariants` passes. An agent that proposes a regressing clause
will assert it without validation. This is the core safety requirement of the meta-loop.
Implement `evaluate_candidate/3` and gate `safe_assert` on `Verdict \= regressed`.

**CRITICAL-2: `check_invariants/1` (verify.pl) is dead code — not called from `safe_assert/1`.**

`safety_bridge.pl:safe_assert/1` calls only `constraints:check_constraints(Clause)`.
It does NOT call `verify:check_invariants/1`. The announced protection against redefining
`safe_assert`, `merkle_commit`, `improve_if_slow`, etc. is unreachable. Wire it in:

```prolog
safe_assert(Clause) :-
    safe_assert_metric('sati_central.prolog.skill_assertion_attempt_total', 1),
    constraints:check_constraints(Clause),   % CHR: banned predicates in body
    verify:check_invariants(Clause),         % ADD THIS: head redefinition guard
    ( current_prolog_flag(test_mode, true)
    -> Proof = proof{...}
    ;  catch(merkle_bridge:merkle_commit(...), ...)
    ),
    assertz(Clause),
    ...
```

**HIGH-1: `head_is_meta` CHR rule does not protect meta-predicate heads.**

The briefing claims: "Added `head_is_meta/1` constraint to `constraints.chr`, throwing a
`safety_violation` if an agent attempts to redefine the meta-improvement logic itself."

This is incorrect. `find_banned/1` scans the *body* of a clause. The `is_meta` check fires
when a meta predicate appears in a body call, not when a new clause's *head* matches a meta
predicate. `safe_assert((improve_if_slow(X,Y) :- write(X)))` passes the CHR check because:
- Head `improve_if_slow(X,Y)` is not scanned by `find_banned`
- Body `write(X)` is not banned and is not meta

The protection announced exists only on paper. `verify:check_invariants/1` IS the correct
protection (it checks the head), but it must be called from `safe_assert/1` (see CRITICAL-2).
The `head_is_meta` CHR constraint is a red herring — it fires for body calls to meta
predicates, which is a policy that may or may not be desired, but it does not prevent
head redefinition. Remove the inaccurate claim from the briefing or fix the CHR rule to
actually check heads.

**HIGH-2: `inject_cut` strategy is semantically broken.**

```prolog
inject_cut((Head :- Body), (Head :- (Body, !)))
```

Adding `!` at the END of Body (`(Body, !)`) does not improve determinism. By the time `!`
executes, Body has already run to completion — the cut only eliminates remaining choice
points for the current predicate, but if the intent is to make a predicate deterministic,
the cut needs to be at the decision point, not after the entire body. For a single-clause
predicate with no alternatives, this is a complete no-op.

Furthermore, the `!` transformation can break predicates used as generators: if
`test_slow_skill(a)` is called inside `findall`, the cut has no effect (cuts are local to
the clause, not to `findall`). If it is called from a context where backtracking is
expected, the cut silently eliminates valid answers.

This strategy produces false confidence — `test_meta.pl` passes because `optimize_skill/1`
forces threshold to 0.0 and the test checks that `(wait_a_bit, !)` is asserted, which it
is. But the transformation itself is either a no-op or harmful.

Replace `inject_cut` with a correct baseline strategy, or remove it and implement the
`propose_improvement/3` skeleton the CLAUDE.md spec requires (which delegates to a proper
analysis of `CurrentClauses`). A stub that does nothing is better than a stub that does
something wrong.
