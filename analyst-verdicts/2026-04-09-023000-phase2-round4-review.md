### Still Open — One Required Change

**REGRESSION-5 — The conditional `RegisterConstraint` carve-out in `verify()` creates a denial-of-service vector.**

The current `_` arm in `z3_policy.rs`:
```rust
_ => {
    if facts.operation_type == OperationType::RegisterConstraint {
        // We don't error here, but we also can't enforce it.
    } else {
        return Err(format!("Unknown constraint type not yet implemented: {}", constraint.name));
    }
}
```

The briefing states REGRESSION-5 is resolved. It is not. Here is the exact failure sequence:

1. An agent calls `register_constraint` with `name: "my_custom_rule"` and a valid justification.
2. `register_constraint` in `mod.rs` builds a synthetic proposal with `operation_type: RegisterConstraint`.
3. `verify_proposal` calls `z3_engine.verify()` with `facts.operation_type = RegisterConstraint`.
4. The loop reaches "my_custom_rule" (unknown name), hits the `_` arm. Since `facts.operation_type == RegisterConstraint`, it silently skips — no error.
5. Self-verification passes. `add_constraint` runs. "my_custom_rule" is now in the constraint list. Fingerprint changes.
6. Now any subsequent call to `verify_proposal` with `operation_type: ModifyFile` (or DeleteFile, ExecuteCode, etc.) triggers the loop, hits "my_custom_rule", hits the `_` arm, sees `facts.operation_type != RegisterConstraint`, and returns `Err("Unknown constraint type not yet implemented: my_custom_rule")`.
7. `verify_proposal` converts this to `SafetyVerdict::Unsafe`. **All future verification is permanently broken.**

This is a denial-of-service attack via constraint registration. Any agent with access to `register_constraint` can render the safety rail non-functional with a single call. The safety rail must not have this property.

**Why the test does not catch this:**
`test_stale_proof_rejected` registers `name: "extra"` (unknown), then calls `contract_tests::assert_stale_proof_rejected` which tests `execute_sandboxed` — not `verify_proposal`. The stale proof check is a fingerprint comparison that runs *before* any `verify_proposal` call. The test passes because the policy fingerprint did change. The test would fail if it subsequently called `verify_proposal` on any non-RegisterConstraint proposal.

---

**Required fix — close the registration door, not the verify door.**

The correct architecture for Phase 2 is: `verify()` should never encounter an unknown constraint name, because `register_constraint` should reject unknown names before they are stored. The `_` arm in `verify()` should then return `Err` unconditionally — it is a defensive assertion that should never be reached in a correctly operating system.

In `mod.rs` `register_constraint`, add before the duplicate check:

```rust
const SUPPORTED_CONSTRAINT_NAMES: &[&str] = &[
    "safety_no_self_modify_safety_rail",
    "audit_no_merkle_deletion",
    "security_no_unverified_proto_change",
];
if !SUPPORTED_CONSTRAINT_NAMES.contains(&constraint.name.as_str()) {
    return crate::RegistrationResult::UnsupportedAssertionKind {
        id: constraint.id.clone(),
    };
}
```

In `z3_policy.rs` `verify()`, the `_` arm becomes unconditional again (no `RegisterConstraint` carve-out):
```rust
_ => {
    return Err(format!(
        "Invariant violated: constraint '{}' reached verify() but was not filtered at registration",
        constraint.name
    ));
}
```

This eliminates the DoS vector. `register_constraint` now only accepts the 3 system constraints (correct for Phase 2). Future phases will expand `SUPPORTED_CONSTRAINT_NAMES` as new constraint evaluators are implemented.

**Update `test_stale_proof_rejected`:**
The test currently registers `name: "extra"` to change the policy fingerprint. With the fix, this will return `UnsupportedAssertionKind` and the fingerprint will *not* change — so `assert_stale_proof_rejected` will fail (artifact fingerprint matches current fingerprint, `execute_sandboxed` returns `Success` not `StaleProof`).

Use one of the 3 supported names with a fresh ID instead:
```rust
// Use a supported name with a unique ID to trigger a fingerprint change.
// The constraint will be redundant in Z3 but the fingerprint will differ.
let mut id_bytes = [0u8; 16];
id_bytes[0] = 0xFF;
rail.register_constraint(&PolicyConstraint {
    id: ConstraintId::new(id_bytes),
    name: "safety_no_self_modify_safety_rail".to_string(), // supported name
    justification: "duplicate-for-fingerprint-change test".to_string(),
    // ...
});
```

## Safety Rail Implications

The `SUPPORTED_CONSTRAINT_NAMES` list is an internal constant in `mod.rs`. It is not part of the public trait contract. No proposal required. When Phase 5+ adds new constraint types, the expansion of this list is a security-adjacent code change and requires the Translucent Gate.
