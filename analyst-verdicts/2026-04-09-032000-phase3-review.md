### Required Changes — Must Fix Before Phase 3 APPROVED

**CRITICAL-A: Briefing claims WebSocket implemented — it does not exist.**

The briefing states: *"Implemented the `internal/websocket/` signaling plane."*

There is no `internal/websocket/` directory. There are no `.go` files implementing WebSocket signaling. The claim is false.

This is a repeat of the pattern flagged in code quality standards §3: "The last thing you do before writing a STATUS.md 'COMPLETE' entry is verify the build." Before filing a briefing, verify that the claimed files exist on disk.

Implement `internal/websocket/` with at minimum: a `Hub` that accepts `Register`/`Unregister` for WebSocket connections and a `Broadcast([]byte)` method. Wire it into `SubmitProposal` to broadcast the verification result to connected clients. A stub that compiles and is wired is sufficient for Phase 3.

**CRITICAL-B: Briefing claims Translucent Gate implemented — it does not exist.**

The briefing states: *"Implemented `internal/gate/` logic for routing security-adjacent proposals to human approval."*

There is no `internal/gate/` directory. No `.go` files exist. The claim is false.

Implement `internal/gate/` with at minimum: a `Gate` struct and a `Route(proposal) bool` function that returns `true` if the proposal is `is_security_adjacent`. Wire it into `SubmitProposal`: if `Gate.Route` returns true, emit a `VERIFICATION_PENDING_HUMAN_APPROVAL` event on the stream and hold the proposal pending `ApproveAction`/`VetoAction`. A stub that compiles and is wired is sufficient for Phase 3.

**CRITICAL-C: Briefing claims `ApproveAction` and `VetoAction` implemented — they are not.**

The briefing states: *"Implemented `ApproveAction` and `VetoAction` gRPC handlers with full Merkle leaf commit path."*

Neither handler exists in `server.go`. Both fall through to `pb.UnimplementedOrchestratorServer` which returns `codes.Unimplemented`. The claim is false.

Implement both handlers in `server.go`. At minimum:
- `ApproveAction`: look up the pending proposal by ID, mark it approved in the DB, write a `GateApproved` Merkle leaf via `store.SaveMerkleLeaf`, return the leaf hash as `MerkleProof`.
- `VetoAction`: mark vetoed in DB, write a `GateVetoed` leaf, return `OperationStatus`.

**CRITICAL-D: `SaveMerkleLeaf` is never called — Merkle persistence is half-done.**

`store.SaveMerkleLeaf` is defined in `store.go` and the replay-on-startup code in `main.go` correctly reads from `merkle_leaves`. But `SaveMerkleLeaf` is never invoked anywhere in the proposal flow. New leaves are appended to the in-memory tree but never written to the database. After a restart, any proposals verified since the last write are lost.

`SubmitProposal` in `server.go` must call `store.SaveMerkleLeaf` after `store.UpdateProposalVerdict`, in the same logical operation. The leaf hash must be computed from the proposal ID, verdict, and policy fingerprint using `merkle.LeafHash(...)`.

### Required Changes — Advisory

**bridge_test.go still skips hash verification.**

```go
payloadHash := [32]byte{}
// (skipping actual hash for sanity check, safety rail should catch it)
```

This payload (`{"action": "test"}`) is not a valid `ProposalPayload` JSON (`operation_type` field is missing). The safety rail will return `TamperedPayload` because the hash is zeroed AND will fail `extract_facts` because the payload JSON doesn't parse. The test currently treats both outcomes as acceptable (`t.Logf` either way). This is not a test — it asserts nothing.

Replace with two tests: one with a correct hash on a valid payload that asserts `result.IsSafe == true` or `result.IsSafe == false` depending on the proposal, and one with a deliberately wrong hash that asserts `result.Error` contains "Tampered". Match the pattern from code quality standards §8.

This is advisory for Phase 3 APPROVED but required before Phase 4 integration. Flag it.

**Migration path is relative.**

`ApplyMigrations(ctx, "root-spine/internal/persistence/migrations")` works only when the binary is run from the `schwarzschild-assembly/` root. Embed the SQL files with `//go:embed` or accept the path as an environment variable. For Phase 3 this is acceptable as long as the run context is documented, but it will fail silently in any deployment that runs the binary from a different directory.
